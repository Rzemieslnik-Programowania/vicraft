# vicraft

**Spec-driven AI workflow CLI for Aider** — part of the [Vitooler](https://github.com/danjarosz) series by [Rzemieślnik Programowania](https://github.com/danjarosz).

vicraft orchestrates a structured, step-by-step development workflow where every change passes through issue → spec → plan → implementation → review before landing in a commit. Aider does the AI work; vicraft does the orchestration.

## Requirements

- [Rust](https://rustup.rs) 1.80+
- [Aider](https://aider.chat) (`pip install aider-chat`)
- [Ollama](https://ollama.com) with `qwen3-coder:30b` (default model) — or any model Aider supports
- [git](https://git-scm.com)
- [gh](https://cli.github.com) (optional, for `vicraft pr`)

## Installation

Install straight from GitHub (no manual clone needed):

```bash
cargo install --git https://github.com/Rzemieslnik-Programowania/vicraft.git
vicraft --version
```

Or clone first and install from the working tree — useful if you plan to hack on vicraft:

```bash
git clone https://github.com/Rzemieslnik-Programowania/vicraft.git
cd vicraft
cargo install --path .
vicraft --version
```

> **Note:** `cargo install --path .` fails with *could not find `Cargo.toml`* if you run it outside the repo root. Make sure `pwd` shows the cloned `vicraft` directory before running it.

`cargo install` places the binary in `~/.cargo/bin/vicraft`. rustup normally adds this directory to your `PATH`; if `vicraft --version` prints *command not found*, append it yourself (zsh):

```bash
grep -qxF 'export PATH="$HOME/.cargo/bin:$PATH"' ~/.zshrc || \
  echo 'export PATH="$HOME/.cargo/bin:$PATH"' >> ~/.zshrc
source ~/.zshrc
```

For bash, replace `~/.zshrc` with `~/.bashrc`.

If you'd rather install system-wide, build manually and copy the binary. Note that if you set `build.target-dir` in `~/.cargo/config.toml`, the binary lives there instead of `./target/`:

```bash
cargo build --release
sudo cp "$(cargo metadata --format-version 1 --no-deps | \
  python3 -c 'import sys,json; print(json.load(sys.stdin)["target_directory"])')/release/vicraft" \
  /usr/local/bin/vicraft
```

## Configuration

Global config at `~/.config/vicraft/config.toml`. Created with defaults on first use.

```toml
[aider]
model = "ollama/qwen3-coder:30b"
extra_flags = ["--no-auto-commits", "--map-tokens", "4096"]

[linear]
api_token = ""   # or export SPEQ_LINEAR_TOKEN
team_id = ""

[context7]
enabled = true

[git]
base_branch = "main"
branch_prefix = "feat/"

[editor]
command = ""     # leave empty to use $VISUAL / $EDITOR
```

Per-project config is in `.aider.conf.yml` (auto-generated by `vicraft init`).

## Workflow

```
STEP 0  vicraft new-issue <name>        create issue from template
STEP 1  vicraft spec <issue-file>       generate spec (AI)
        ↳ answer Open questions in spec before proceeding
STEP 2  vicraft plan <spec-file>        generate plan (AI) — blocked if Open questions unanswered
STEP 3  vicraft impl <plan-file>        Aider implements → WIP commit created automatically
        vicraft clear-context           reset Aider history before review
STEP 4  vicraft review                  AI reviews diff vs spec + plan
        ↳ if Approved  →  vicraft commit  →  vicraft pr
        ↳ if changes   →  vicraft plan <review-file>  →  back to STEP 3
```

## Commands

| Command | Description |
|---|---|
| `vicraft init` | Initialize project structure (`.aider/`, `.issues/`, etc.) |
| `vicraft scan` | Analyze codebase → `.aider/context/` |
| `vicraft new-issue <name>` | Create `.issues/<date>-<slug>.md` from template |
| `vicraft new-issue <name> --open` | Same, opens in `$EDITOR` |
| `vicraft spec <file\|LINEAR-ID>` | Generate spec from issue [STEP 1] |
| `vicraft plan <spec-file>` | Generate implementation plan [STEP 2] |
| `vicraft plan <review-file>` | Regenerate plan from review (iteration) |
| `vicraft impl <plan-file>` | Run Aider implementation + WIP commit [STEP 3] |
| `vicraft review` | AI review of diff vs spec/plan [STEP 4] |
| `vicraft commit` | Amend WIP commit with conventional message |
| `vicraft commit --staged` | New commit from staged files |
| `vicraft pr` | Create GitHub PR via `gh` |
| `vicraft clear-context` | Delete Aider chat history |
| `vicraft skills list` | List skill files |
| `vicraft skills new <name>` | Create a new skill file |
| `vicraft skills edit <name>` | Edit a skill file |

## Project structure (in your target project)

```
your-project/
├── .aider/
│   ├── CONVENTIONS.md          ← fill in after init
│   ├── skills/                 ← project knowledge base
│   │   ├── SKILL.architecture.md
│   │   ├── SKILL.database.md
│   │   └── ...
│   ├── context/                ← auto-generated, gitignored
│   │   ├── CODEBASE.md
│   │   ├── DEPENDENCIES.md
│   │   └── PATTERNS.md
│   └── templates/              ← all four templates
├── .issues/                    ← task descriptions
├── .specs/                     ← generated specs
├── .plans/                     ← generated plans
├── .implementations/           ← impl summaries
└── .reviews/                   ← AI review files
```

All artifact directories (`.issues/`, `.specs/`, `.plans/`, `.implementations/`, `.reviews/`) are committed to the repository — they document decisions. Only `.aider/context/` is gitignored.

## WIP commit strategy

`vicraft impl` always runs Aider with `--no-auto-commits` to prevent Aider from polluting git history. After Aider finishes, vicraft creates a single `wip: <task-id>` commit. On iteration (STEP 5), the same WIP commit is amended rather than duplicated. `vicraft commit` then amends it once more with the final conventional commit message — giving you one clean commit per task.

## Linear integration

Set `linear.api_token` in config (or `SPEQ_LINEAR_TOKEN` env var) to fetch issues directly:
